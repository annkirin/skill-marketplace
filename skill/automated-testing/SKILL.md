# Automated Testing Skill

在每次代码变更完成后，执行以下流程确保代码质量：

## 第一阶段：自动化测试

### 第一步：识别测试框架

首先识别项目使用的测试框架和包管理器：

- 检查 `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod` / `Makefile` 等
- 确认测试命令（如 `npm test`、`pytest`、`cargo test`、`go test`）
- 确认测试文件位置（`__tests__/`、`tests/`、`*.test.*`、`*_test.*`）

### 第二步：运行测试

```bash
npm test  # 全量测试
```

记录结果：
- 全部通过 → 进入第二阶段
- 有失败 → 进入第三步

### 第三步：分析并修复失败

对每个失败的测试：

1. **阅读失败信息**：断言期望 vs 实际值，错误堆栈
2. **定位根因**：
   - 是新代码引入的 bug？
   - 是测试期望需要更新？（接口签名变了但测试没跟上）
   - 是环境/依赖问题？
3. **修复**：
   - 如果是代码问题 → 修复实现代码
   - 如果是测试过时 → **谨慎更新测试**，确保新期望合理
4. **重跑验证**：确认通过

### 第四步：回归验证

```bash
npm test  # 全量测试确认无回归
```

### 第五步：补充缺失测试

- 新增的函数/类/模块 → 必须有对应测试
- 修改的逻辑分支 → 检查是否覆盖了所有分支
- 边界条件 → 空值、极值、异常输入

## 第二阶段：模拟人类实操验证

> 以真实用户视角，模拟完整的操作链路，验证代码在真实场景中是否能正常工作。

### 1. 核心用户流程验证

梳理本次变更涉及的核心用户路径，逐一模拟操作：

| 场景 | 模拟步骤 | 预期结果 |
|------|---------|---------|
| 正常流程 | 按用户典型路径从头到尾走一遍 | 每一步成功，数据正确 |
| 快捷路径 | 用快捷键/短链接/默认行为操作 | 行为正确，无报错 |
| 反向操作 | 增→删、创建→取消、开→关 | 状态回退正确，无残留 |

示例：
```
场景：用户注册 → 登录 → 创建文章 → 发布
操作：模拟填写表单、点击提交、检查页面跳转、核对数据库数据
```

### 2. 异常输入验证

模拟用户各种"不按套路出牌"的操作：

- **空值**：必填字段留空提交
- **越界**：超长文本、超大数字、特殊字符、XSS/脚本注入
- **格式错误**：邮箱填文字、数字填字母、日期填乱码
- **重复操作**：快速双击提交、连点保存按钮
- **中断操作**：中途刷新页面、后退前进、关闭弹窗

对于 Web / API 项目，直接用 curl / httpie 模拟请求：
```bash
# 空值测试示例
curl -X POST http://localhost:3000/api/register -d '{"email":"","password":""}'

# 越界测试
curl -X POST http://localhost:3000/api/register \
  -d '{"email":"a@b.com","password":"'"$(python -c 'print("A"*10000)')"'"}'

# 特殊字符/注入
curl -X POST http://localhost:3000/api/search \
  -d '{"q":"<script>alert(1)</script>; DROP TABLE users;"}'
```

检查响应：应该返回 400/422 和友好的错误信息，而不是 500 崩溃。

### 3. 状态与边界验证

- **权限边界**：未登录用户能否访问需要登录的页面/API？
- **数据边界**：列表为空时、只有一条时、满页时 → UI 是否正常？
- **并发操作**：两个用户同时编辑同一条数据 → 会不会互相覆盖？
- **幂等性**：重复提交相同请求 → 不会产生重复数据

### 4. 环境与兼容性

如果项目有前端 / 客户端：

- **浏览器兼容**：Chrome 和标准浏览器下渲染是否正常
- **响应式**：不同屏幕尺寸下布局是否错乱
- **加载状态**：网络慢时是否有 Loading、骨架屏，而不是白屏
- **错误展示**：接口失败时是否显示友好提示，而非崩溃白页

### 5. 数据库/持久化验证

- **CRUD 验证**：创建的数据能查出来吗？修改后能查到新值吗？删除后真的没了吗？
- **级联影响**：删除父记录后，关联的子记录是否按预期处理（级联/置空/阻止）？
- **数据一致性**：事务范围内，失败时是否回滚？不会出现脏数据？
